# Formidable POST

_Let's see what the browser POSTs when it thinks no one is looking._

**Goal**

Capture and analyze the HTTP POST request generated by submitting a simple web form, focusing on headers, body encoding, and predicting payload changes.

**Instructions**

1. **Setup:** Use Wireshark and Browser DevTools. Target the form at `http://httpbin.org/forms/post`. Sometimes website id down for a few minutes - you need to wait a while.
2. **Action:** Capture the process of submitting the form with sample data via your browser.
3. **Analysis:**
    - Locate the resulting HTTP `POST` request in both tools.
    - Analyze the request body format.
    - **Question:** How would the payload change if the form included an additional `Delivery Tip` field?

**Submission**

1. A screenshot showing the **request body/payload** for the POST request as displayed in Wireshark
2. A screenshot showing the **request body/payload** for the POST request as displayed in your Browser's Developer Tools.
3. Your written answer to the question.

<aside>

Hint:

If Safari tries to use HTTPS you need to force it to connect over HTTP, do this:

1. Open **Safari → Preferences → Advanced** and enable **“Show features for developers”**
2. In the menu bar, go to **Develop → Settings → Disable Local Overrides** (just to be sure nothing’s cached).
3. Then manually type the full **http://** address in the URL bar — for example: [`http://httpbin.org/get`](http://httpbin.org/get)
4. Don’t press Enter too early; make sure Safari doesn’t auto-replace it with HTTPS before loading.
5. If Safari still switches to HTTPS automatically, open **Terminal** and run: `open -a Safari "<http://httpbin.org/get>"`
6. That command will force Safari to open the page via plain HTTP. </aside>

But it’s better to use Chrome. In **Chrome** make sure you have “Secure Connections” turned off in Settings > Privacy and security:
